HTTP Semantics vs Syntax

Một cách nhìn đơn giản: HTTP vẫn nói cùng một “ngôn ngữ ý nghĩa”, nhưng cách đóng gói để gửi qua mạng đã thay đổi rất nhiều.

Tóm tắt trong một câu

Semantics là “HTTP muốn diễn đạt điều gì”. Syntax là “HTTP được viết hoặc đóng gói thành bytes như thế nào khi đi trên mạng”.

Semantics = Ý nghĩa

Ứng dụng đang làm gì?

  • GET hay POST?
  • Đường dẫn nào?
  • Status code là gì?
  • Header và cookie nói gì?
  • Body chứa dữ liệu gì?
≠
Syntax = Cách biểu diễn

Nó được gửi đi như thế nào?

  • Dạng text dễ đọc?
  • Dạng binary frame?
  • Chạy trên TCP?
  • Chạy trên QUIC/UDP?
  • Có nén header hay multiplexing?

Mô hình dễ nhớ

Semantics HTTP message có ý nghĩa gì? Method, path, status code, header, body
Syntax Ý nghĩa đó được mã hóa ra sao? Text lines, binary frames, pseudo-headers
Transport Bytes đi qua mạng bằng gì? TCP hoặc QUIC trên UDP
Web application nhìn thấy ý nghĩa HTTP. Network stack xử lý cách truyền bytes.

Ví dụ chi tiết: cùng một request, nhiều cách biểu diễn

HTTP/1.1: text-based, dễ đọc bằng mắt
POST /api/login HTTP/1.1
Host: shop.example.com
User-Agent: Browser/120
Content-Type: application/json
Cookie: tracking_id=abc
Content-Length: 48

{
  "username": "alice",
  "password": "P@ssw0rd!"
}

Dễ thấy request line, header và body. Đây là hình ảnh “kinh điển” khi học HTTP.

HTTP/2 hoặc HTTP/3: semantics giống, syntax khác
:method POST
:scheme https
:authority shop.example.com
:path /api/login
user-agent Browser/120
content-type application/json
cookie tracking_id=abc

DATA frame:
{
  "username": "alice",
  "password": "P@ssw0rd!"
}

On the wire:
binary frames / QUIC streams

Ý nghĩa vẫn là POST /api/login với JSON body, nhưng wire format không còn giống HTTP/1.1 text thuần.

Phản hồi cũng vậy

HTTP/1.1 Response
HTTP/1.1 200 OK
Content-Type: application/json
Set-Cookie: session=xyz; HttpOnly; Secure
Content-Length: 36

{
  "role": "user",
  "login": true
}
HTTP/2 / HTTP/3 Response
:status 200
content-type application/json
set-cookie session=xyz; HttpOnly; Secure

DATA frame:
{
  "role": "user",
  "login": true
}
Điểm cần nhìn ra: Với kiểm thử Web, các câu hỏi quan trọng vẫn giống nhau: endpoint nào được gọi, method gì, cookie/token nào được gửi, body chứa gì, server trả status code nào, response có lộ dữ liệu không. Đây là phần semantics.

HTTP đời đầu nhìn như một giao thức nguyên khối

Góc nhìn ban đầu

Request/Response: GET, status code, header, body
Cú pháp: dòng text, dấu CRLF, header dạng key/value
Truyền tải: gửi bytes qua TCP connection

Khi học HTTP/1.x, các phần này thường xuất hiện cùng nhau, nên HTTP dễ bị nhìn như một khối duy nhất.

Góc nhìn hiện đại

Semantics giữ tương đối ổn định: method, status, header, body
Syntax thay đổi: text → binary framing → QUIC streams
Transport thay đổi: TCP trong HTTP/1.1, HTTP/2; QUIC/UDP trong HTTP/3

Cách truyền thay đổi để tối ưu hiệu năng, nhưng ý nghĩa HTTP mà ứng dụng xử lý vẫn quen thuộc.

Timeline ngắn

HTTP/0.9
Semantics: rất đơn giản, chủ yếu GET tài liệu.
Syntax: text rất tối giản, chưa có hệ header/status phong phú như sau này.
HTTP/1.1
Semantics: method, status code, header, body trở thành nền tảng quen thuộc.
Syntax: text-based, dễ đọc, dễ demo bằng telnet/netcat/Burp.
HTTP/2
Semantics: vẫn là request/response, method, path, header, body.
Syntax: binary framing, multiplexing, header compression.
HTTP/3
Semantics: HTTP semantics vẫn tiếp tục được dùng.
Syntax/Transport: chạy trên QUIC, QUIC chạy trên UDP.

Bảng so sánh nhanh

Thứ cần phân biệt Ví dụ Ý nghĩa khi phân tích HTTP
Semantics GET, POST, 200, 403, Cookie, Authorization, JSON body Cho biết ứng dụng đang yêu cầu gì, server trả lời gì, dữ liệu nào đang được trao đổi.
Syntax HTTP/1.1 text lines, HTTP/2 binary frames, HTTP/3 QUIC streams Cho biết cùng một ý nghĩa đó được biểu diễn thành bytes như thế nào trên mạng.
Transport TCP hoặc QUIC trên UDP Ảnh hưởng đến hiệu năng, kết nối, multiplexing, latency, nhưng không làm biến mất các khái niệm HTTP cốt lõi.
Kết luận: Khi kiểm thử ứng dụng Web, đừng bị rối bởi phiên bản HTTP. Hãy bắt đầu từ semantics: method, path, header, cookie, body, status code. Sau đó mới quan tâm syntax/transport nếu cần phân tích sâu hơn về HTTP/2, HTTP/3 hoặc vấn đề ở tầng giao thức.
